## 2. 项目背景

本项目面向"AI 语音绘图工具"议题开发。议题要求用户不能使用鼠标或键盘，仅通过语音指令完成绘图创作。

因此，本项目的核心不是做一个普通画板，也不是做一个 AI 文生图工具，而是实现一个"语音指令驱动的可控绘图系统"。

用户通过语音表达绘图意图，系统识别语音内容，将自然语言转换为结构化绘图 action，再由 Canvas 绘图引擎执行操作，最后通过文字和语音反馈告诉用户执行结果。

核心交互链路：

```text
用户说话
  ↓
浏览器语音识别
  ↓
指令解析
  ↓
生成结构化 DrawingAction
  ↓
Canvas 执行绘图操作
  ↓
系统语音反馈
  ↓
继续监听下一条指令
```

本项目需要同时体现：

1. 语音交互能力
2. 指令理解能力
3. 绘图执行能力
4. 复杂指令拆解能力
5. 容错与澄清能力
6. 成本控制意识
7. 工程化开发流程

## 3. 一句话介绍

Say2Draw 是一款语音驱动的可编辑矢量画板。底层使用基础图形，LLM 能像厨师一样将它们组合成有对象感、有构图的画面。它不是一次性生图，而是可编辑、可迭代、可控制的画。

## 4. 核心问题定义

### 4.1 用户问题

传统绘图工具通常依赖鼠标、键盘、触控板或触摸屏操作。对于只想快速表达视觉想法的用户来说，这类工具存在三个问题：

1. 操作门槛高：用户需要先学习工具按钮、菜单和快捷键。
2. 表达不自然：用户心里想的是"画一个太阳"，但工具要求用户一步步选择形状、颜色和位置。
3. 不适合无手操作场景：当用户双手不方便时，传统绘图工具几乎无法使用。

### 4.2 产品问题

本项目尝试回答：

> 用户能否仅通过自然语言语音指令，完成一个可控、可编辑、可撤销的绘图过程？

这里的关键不是"生成一张漂亮图片"，而是实现一条可靠的执行链路：

1. 系统听懂用户在说什么。
2. 系统判断这句话对应什么绘图操作。
3. 系统把自然语言转成可执行的结构化 action。
4. 系统执行 action，并让结果可见。
5. 系统在听懂时不乱执行，而是提示用户补充信息。

---

### 4.3 差异化特性

与现有 AI 绘图方案相比，Say2Draw 的核心差异化在于：

#### 4.3.1 累积式绘图

每一步语音指令都在画布上留下独立的图形元素。用户说"画一个太阳"，太阳出现；说"加两朵云"，云出现，太阳仍在原位。

对比：AI 文生图工具每次指令都会重新生成整张图片，无法累积。

#### 4.3.2 可中断

用户说到一半可以停下来，系统会处理已识别的部分，不需要等待完整指令。

对比：AI 文生图工具需要完整的 prompt 才能开始生成，无法中途打断。

#### 4.3.3 可局部修改

用户可以撤销、删除或修改单个图形元素，而不影响其他元素。

对比：AI 文生图工具无法修改生成图片的局部，只能重新生成。

#### 4.3.4 实时执行

Canvas 在用户说话的同时立即渲染图形，不需要等待 AI 生成。

对比：AI 文生图工具通常需要数秒到数十秒的生成时间。

## 7. 使用场景

### 7.1 基础绘图场景

用户打开应用后，系统自动进入监听状态。

用户说：

```text
画一个红色圆形
```

系统执行：

1. 识别语音文本。
2. 解析出颜色：红色。
3. 解析出图形：圆形。
4. 生成 add_shape action。
5. 在 Canvas 上绘制红色圆形。
6. 播报："已为你画了一个红色圆形。"

### 7.2 添加文字场景

用户说：

```text
写上你好
```

系统执行：

1. 识别文字内容"你好"。
2. 生成 text shape。
3. 在 Canvas 上添加文字。
4. 播报："已添加文字：你好。"

### 7.3 撤销场景

用户说：

```text
撤销
```

系统执行：

1. 识别撤销指令。
2. 回退到上一个画布状态。
3. 播报："已撤销上一步操作。"

如果没有可撤销操作，系统播报：

```text
当前没有可以撤销的操作。
```

### 7.4 清空画布场景

用户说：

```text
清空画布
```

系统执行：

1. 删除所有图形。
2. 更新画布。
3. 播报："画布已清空。"

### 7.5 复杂指令拆解场景

用户说：

```text
画一个太阳、两朵云和一棵树
```

系统执行：

1. 本地规则解析失败。
2. 进入 LLM 结构化解析。
3. LLM 返回 batch_actions。
4. 系统依次绘制太阳、云朵和树。
5. 操作历史展示拆解结果。
6. 播报："已根据你的描述绘制了多个元素。"

### 7.6 模糊指令澄清场景

用户说：

```text
画一个好看的东西
```

系统执行：

1. 判断指令缺少明确图形对象。
2. 不直接绘图。
3. 返回 ask_clarification。
4. 播报："我还不确定你想画什么，可以告诉我是圆形、矩形、线条、文字，还是一个具体物体吗？"

## 8. 核心设计原则

### 8.1 纯语音优先

本项目必须符合议题要求：

> 用户不能使用鼠标或键盘，仅通过语音指令完成绘图创作。

因此：

1. 应用启动后应自动尝试进入语音监听状态。
2. 不把"点击开始监听"作为核心交互路径。
3. 浏览器首次麦克风授权可以接受，但应用内绘图操作必须通过语音完成。
4. 开发阶段可以保留 dev-only 测试按钮，但最终提交前必须隐藏或明确标注。
5. Demo 视频必须展示通过语音完成核心绘图操作。

### 8.2 稳定性优先

72 小时内不追求大而全，优先保证核心链路稳定：

```text
语音 → 文本 → action → Canvas → 反馈
```

只要这条链路稳定，作品就是有效的。

### 8.3 本地解析优先

常见指令必须优先使用本地规则解析。

例如：

1. 画一个红色圆形
2. 画一个蓝色矩形
3. 画一条黑色线
4. 写上你好
5. 清空画布
6. 撤销

这些指令不需要调用 LLM。

这样可以：

1. 降低 API 成本。
2. 减少响应延迟。
3. 提高稳定性。
4. 展示工程取舍能力。

### 8.4 LLM 只做兜底

LLM 用于处理复杂自然语言指令，不用于所有指令。

LLM 只在以下场景使用：

1. 本地规则解析失败。
2. 指令包含多个对象。
3. 指令表达更自然，但仍可转化为绘图 action。
4. 需要拆解复杂意图。

LLM 必须返回结构化 JSON，不允许直接返回自然语言后执行。

### 8.5 不乱执行

系统听不懂时，不应该随便画。

优先级：

```text
准确执行 > 澄清追问 > 拒绝执行 > 乱猜乱画
```

## 9. 功能范围

## 9.1 P0：必须实现

P0 是保证作品有效和 Demo 可跑的最低范围。

### P0-01：自动语音监听

功能说明：

应用加载后自动尝试进入语音监听状态，不把"点击开始监听"作为核心路径。

需求：

1. 检测浏览器是否支持 SpeechRecognition。
2. 页面加载后自动尝试启动识别。
3. 使用 continuous 模式。
4. 使用 interimResults 展示中间识别结果。
5. 显示当前语音状态。

语音状态至少包括：

1. booting：启动中
2. permission_required：需要麦克风权限
3. listening：监听中
4. processing：处理中
5. speaking：系统播报中
6. error：错误
7. unsupported：浏览器不支持

验收标准：

1. 页面加载后系统自动尝试进入监听。
2. 用户不需要通过鼠标或键盘触发绘图指令。
3. 识别到 final result 后触发指令处理。
4. 浏览器不支持时显示提示。
5. 麦克风权限异常时显示提示。

### P0-08：回声保护

功能说明：

系统语音播报可能被语音识别模块捕获，因此必须避免系统把自己的反馈当成用户指令。

方案：

1. 使用 isSpeakingRef 标记系统正在播报。
2. 播报期间忽略识别结果。
3. 播报结束后恢复监听。
4. 必要时在 recognition.onend 中重启监听。

验收标准：

1. 系统播报不会触发重复绘图。
2. 播报结束后可以继续识别用户指令。
3. 回声保护逻辑不会阻塞正常用户输入。

## 9.2 P1：优先实现

P1 是提升作品竞争力和面试表达空间的功能。

### P1-01：LLM 复杂指令解析

功能说明：

当本地规则解析失败时，调用 LLM 进行结构化解析。

LLM 供应商不锁定单一服务，要求兼容：

OpenAI-compatible structured output API

要求：

1. 本地解析失败后才调用 LLM。
2. LLM 必须返回结构化 JSON。
3. 返回结果必须校验。
4. 非法 action 不执行。
5. API key 不暴露在前端。
6. `.env` 不提交。

验收标准：

1. 简单指令不调用 LLM。
2. 复杂指令进入 LLM 解析。
3. LLM 能返回 add_shape 或 batch_actions。
4. 返回结果通过 schema 校验后才执行。
5. 错误时有清晰反馈。

---

### P1-02：复杂指令拆解

功能说明：

支持用户一次说出多个绘图对象。

示例：

```text
画一个太阳、两朵云和一棵树
```

系统应拆解为多个 action。

验收标准：

1. 支持 batch_actions。
2. 多个 action 按顺序执行。
3. 操作历史展示拆解结果。
4. 单次 batch actions 数量有上限，避免成本和执行风险。

---

### P1-03：模糊指令澄清

功能说明：

当指令无法可靠执行时，系统不乱画，而是询问用户。

示例：

用户说：

```text
画一个东西
```

系统回应：

```text
我还不确定你想画什么，可以告诉我是圆形、矩形、线条、文字，还是一个具体物体吗？
```

验收标准：

1. 能识别模糊指令。
2. 返回 ask_clarification。
3. 不执行不可靠绘图。
4. 系统反馈清楚。

---

### P1-04：重复指令缓存

功能说明：

对于相同复杂指令，可以缓存 LLM 解析结果，减少重复调用成本。

验收标准：

1. 相同文本命中缓存。
2. 缓存结果仍需校验。
3. README 或 DESIGN 中说明该策略。

---

### P1-05：redo 重做操作

功能说明：

支持用户说"重做"，恢复被撤销的操作。

验收标准：

1. undo 后可以 redo。
2. 新增操作后 future 清空。
3. 无可重做状态时有提示。

## 11.4 后续迭代方向

### 对象分解通用原则

LLM 不依赖预存菜谱，而是掌握通用的对象分解能力。LLM 本身就是库，靠推理而非查表：

- 将任意对象拆解为 2-5 个基础图形
- 先画主体（最大图形），再画细节（较小图形）
- 用位置表达关系（眼睛在脸内、屋顶在墙上、光线围绕中心）
- 变化尺寸（主体大、细节小）

示例（不是预存菜谱，是推理能力）：
- 太阳：核心圆 + 光芒线 + 光晕圆
- 树：树干矩形 + 1-3 个重叠树冠圆
- 房子：墙面矩形 + 屋顶矩形 + 门矩形
- 笑脸：大脸圆 + 两个小眼圆 + 弧线嘴
- 猫：圆脸 + 三角耳 + 胡须线
- 花：中心圆 + 环绕花瓣 + 茎线

### 随机扰动（jitter）

所有坐标和尺寸加 ±3px 随机偏移，消除机器规整感，产生手绘般的自然变化。

### 构图提示

教 LLM 不要把元素全堆在画布正中央。太阳放右上角，地面在底部，树在左侧偏下。

### 能力矩阵

| 优先级 | 能力 | 说明 |
|--------|------|------|
| P0 | 对象分解通用原则 | LLM 自主推理如何拆解任意对象 |
| P1 | 场景构图 | 多个对象有整体布局关系 |
| P1 | 随机扰动 | 坐标±3px 消除机器感 |
| P2 | 二次编辑 | 语音修改已画元素 |
| P2 | 多对象布局 | 左右、上下、远近关系 |

## 12. 成本控制策略

题目要求综合考虑成本控制，本项目从产品和工程两方面控制成本。

### 12.1 本地规则优先

常见指令在本地解析，不调用 LLM。

收益：

1. 零 API 成本。
2. 响应更快。
3. 更稳定。
4. 离线可部分使用。

### 12.2 LLM 兜底

LLM 只处理复杂指令。

触发条件：

1. 本地规则解析失败。
2. 指令包含多个对象。
3. 指令需要语义拆解。
4. 指令可以转化为有限 action。

### 12.3 结构化输出

LLM 必须返回结构化 JSON，减少自然语言回复带来的二次解析成本。

### 12.4 Action 数量限制

单次复杂指令最多生成有限数量 action，例如 5 到 8 个。

避免：

1. 画布过载。
2. API 成本不可控。
3. Demo 中执行过慢。
4. 模型生成不必要元素。

### 12.5 重复指令缓存

相同复杂指令可复用解析结果，减少重复调用。

### 12.6 不上传音频

项目优先使用浏览器端语音识别，只把识别后的文本交给 parser 或 LLM。

收益：

1. 降低音频传输成本。
2. 降低隐私风险。
3. 降低后端复杂度。

### 12.7 Canvas 本地渲染

所有图形在浏览器端 Canvas 渲染，不需要服务端生成图片。

收益：

1. 降低服务端计算成本。
2. 提高响应速度。
3. 简化部署。

### 12.8 不做 AI 文生图

AI 文生图成本较高，而且不符合"语音控制绘图工具"的核心要求。本项目选择结构化绘图 action，而不是图片生成。

## 17. 主要风险与应对

### 17.1 Web Speech API 兼容性风险

风险：

不同浏览器对语音识别支持不一致。

应对：

1. README 中明确推荐 Chrome。
2. 页面中检测 unsupported 状态。
3. 不支持时给出清晰提示。
4. Demo 使用兼容浏览器录制。

### 17.2 自动监听被浏览器限制

风险：

浏览器可能要求用户先授予麦克风权限，或限制自动播放/自动监听。

应对：

1. 将首次麦克风授权解释为浏览器安全权限。
2. Demo 前提前授予权限。
3. 页面中显示 permission_required 状态。
4. 应用内绘图流程保持纯语音。

### 17.3 系统播报被误识别

风险：

系统说"已画出红色圆形"时，语音识别可能捕获这段反馈。

应对：

1. isSpeakingRef 标记播报状态。
2. 播报期间忽略识别结果。
3. 播报结束后恢复监听。

### 17.4 LLM 输出不可控

风险：

LLM 返回自然语言、不合法 JSON 或过多 action。

应对：

1. 使用 Structured Outputs。
2. 使用 schema 校验。
3. 限制 action 数量。
4. 非法结果不执行。
5. fallback 到澄清提示。

### 17.5 72 小时时间不足

风险：

功能范围过大导致核心链路不稳定。

应对：

1. P0 优先。
2. P1 只做最小版本。
3. P2 明确不影响提交。
4. 每个 PR 保持主分支可运行。

## 19. PRD 结论

Say2Draw 的 MVP 不追求成为专业绘图软件，而是要在 72 小时内完成一个可信、稳定、可演示的语音绘图系统。

5. 用 PRD、PLAN、SYSTEM、AGENTS 和 TASKS 展示工程化过程。
