# DESIGN.md — Say2Draw 设计文档

> 记录 Say2Draw 的产品定位演变、计划能力 vs 已实现能力对照、技术决策与项目反思。

---

## 1. Product Vision & Philosophy

### 1.1 起点：纯语音控制的 AI 绘图工具

Say2Draw 的最初定位是一个纯语音控制的 AI 绘图工具。用户说出绘图意图，系统将自然语言解析为结构化绘图操作，在 Canvas 上逐个执行。与 AI 文生图（Midjourney、DALL·E）的关键区别在于：Say2Draw 的执行单位是"单个图形"而非"整张图片"，用户可以撤销、修改、累积，而不是每次推倒重来。

### 1.2 转折：价值危机与定位天花板归因

在 PR16 至 PR18 的密集工程推进中，产品暴露出根本问题：用户说不清楚要画什么，系统也画不出用户模糊想象中的东西。最初的反应是"工程不够好"——线条不够细致、构图不够聪明。但深入排查后发现，真正的问题是**产品定位的天花板**，而非工程质量：在任何当前的多模态模型能力边界内，"你说一句，AI 准确画出来"这个预期本身就是不现实的。不是 Say2Draw 画得不够好，而是这个品类在当前技术条件下没有好的解法。

### 1.3 重构：系统谦逊，草图起步，一起打磨

归因完成后，产品定位从"AI 替你画好"转向"**系统谦逊 → 草图起步 → 一起打磨**"：

- **系统谦逊**：AI 公开承认自己可能理解偏差，从不假装有把握
- **草图起步**：任何指令都先给出可见的保守结果，让用户有东西可看可说
- **一起打磨**：用户用自然语言逐轮调整，每次只做一个小改动

这一理念的技术实现是 PR17 引入的 **SketchAgent 统一笔触管线**：LLM 在 50×50 逻辑网格上描述笔触坐标，前端用最小二乘 Bézier 曲线拟合渲染为手绘风格 SVG。LLM 负责语义理解（"画什么"），确定性渲染负责视觉品质（"怎么画"）——两个子系统各自在擅长领域做到极致。

### 1.4 72 小时 MVP 的取舍原则

| 原则 | 含义 |
|---|---|
| P0 必达 | 语音识别 + 基础绘图 + 语音反馈闭环必须完整可演示 |
| P1 尽力 | LLM 增强、计划确认流、多模式场景在时间允许下完成 |
| P2 可弃 | 保存图片、演示模式、移动端适配等明确标记为时间不足则放弃 |
| 主分支始终可运行 | 任何时候 `npm run dev` 启动的应用都能完成核心语音绘图流程 |
| PR 不堆砌 | 每个 PR 只做一件事，禁止最后一天一次性导入代码 |

### 3.4 数据流：一次典型语音绘图的完整路径

```
1. 用户说"画一个小女孩站在月亮下面"
2. Web Speech API → onFinalTranscript(rawTranscript, metadata)
3. 转录审查门：检查 confidence，检查"换/话/化"误识别
4. 审查通过 → 路由判断：无 pendingPlan 无 sketchMode → routeCommands()
5. routeCommands 返回 [{ type: 'generate_sketch' }]
6. handleGenerateSketch(transcript) → POST /api/sketch-plan
7. 服务端调用 Mimo → 返回 { plan: { previewText, elements[], drawingOrder[], ... } }
8. setPendingPlan(plan) — 显示 PlanConfirmOverlay
9. 用户说"确认" → executeApprovedPlan()
10. POST /api/sketch → Mimo 生成 50×50 网格笔触 XML
11. parseSketchXML() → 提取 <strokes> 块
12. fitAndRenderStrokes() → Bézier 曲线拟合 + 网格→像素映射
13. setSketchStrokes(rendered) → SketchLayer 渲染 SVG
14. updateFeedback({ result: "已为你画了...", status: 'success' })
15. SpeechSynthesis 播报反馈 → FeedbackPanel 展示结果
```

## 4. Key Technical Decisions

### 4.1 供应商去风险：DeepSeek → Mimo

**上下文**：PR9-PR16 期间使用 DeepSeek 作为 LLM provider，频繁遇到 `structured output` 兼容性问题，JSON 返回格式不稳定。

**决策**：PR17 全面转向 Mimo 多模态模型。选择标准：(1) OpenAI-compatible 接口，无需改客户端代码；(2) 原生支持图片输入，可直接用于视觉编辑；(3) 手绘笔触输出质量优于 DeepSeek。

**影响**：架构迁移涉及 prompt 重写、50×50 网格坐标适配、错误处理策略调整。

### 4.2 本地解析 vs LLM 解析的分工

**上下文**：所有指令都走 LLM 调用成本高、延迟大；全部本地解析则复杂场景无法覆盖。

**决策**：两层分工——基础指令（颜色+形状、清空、撤销、本地即时调整）走本地正则匹配，零 API 成本、零延迟；复杂指令（场景描述、流程图、视觉编辑）走 LLM。PR18 进一步将本地 parser 缩减为仅处理 clear/undo，其余交给 LLM 统一处理。

### 4.3 SketchAgent 管线：LLM 语义 + 确定性渲染

**上下文**：早期方案让 LLM 直接输出 Canvas 绘图指令（像素坐标+图形参数），结果是坐标混乱、重叠严重。

**决策**：PR17 引入 SketchAgent 两阶段管线。LLM 在 50×50 逻辑网格上输出笔触描述（坐标序列 + t_values），前端用最小二乘 Bézier 曲线拟合 + 网格→像素映射做确定性渲染。LLM 只做语义理解，不做坐标精算。这是"系统谦逊"的工程表达。

### 4.4 流程图：LLM 语义解释 + 确定性布局

**上下文**：PR24 之前，流程图布局完全由 LLM 在 prompt 中决定，结果不稳定——同样的 4 步流程三次画出来位置都不同。

**决策**：PR24 将流程图拆成两个阶段。LLM 输出**结构化意图**（每个节点的 label、相对顺序、连接关系），前端用确定性布局算法计算节点坐标。LLM 不再画流程图，只是说"这个节点在哪个之后、和哪个并列"。

### 4.5 计划→确认→执行三段式交互

**上下文**：早期流程是"语音→LLM→直接画"，用户说完就出图，没有确认环节。问题：复杂场景下 LLM 理解偏差的概率高，直接画出错画布需要"撤销"重来。

**决策**：PR19-21 引入计划→确认→执行三段式。系统先展示绘图计划（元素列表+绘制顺序），用户确认后再执行。用户也可以在确认前说"再详细一点"来修订计划。这个设计降低了出错成本，也给了用户可见的控制感。

### 4.6 PR24/PR25 架构级拆分

**上下文**：PR24 同时涉及流程图编辑逻辑（模型层）和反馈面板显示（视图层）。这两个改动在概念上正交——模型层改动影响服务端+数据流，视图层改动影响前端组件。

**决策**：拆成两个独立 PR。PR24 只改模型层（流程图编辑模型+恢复保护），PR25 只改视图层（反馈面板+状态栏）。拆分后每个 PR 可以独立测试、独立回退，互不污染。

## 5. What We Learned

### 5.1 价值危机的真正原因不是工程质量，而是定位天花板

Say2Draw 在 PR16-PR18 期间经历了一次强烈的价值质疑。最初的归因是"工程不够好"——线条不够细致、构图不够聪明、LLM 输出不够稳定。但逐个修复这些工程问题后，产品体验没有根本改善。

真正的问题是**品类天花板**：在当前多模态模型的能力边界内，"你说一句，AI 准确画出来"这个产品承诺本身就不成立。用户说不清楚要画什么（"画一个宁静的夜晚"到底要画什么？），AI 也画不出用户模糊想象中的画面。这不是 Say2Draw 画得不够好，而是这个品类在当前技术条件下只能到这个程度。

**教训**：工程投入之前，先诊断问题属于"工程质量问题"还是"产品定位问题"。两者的修复策略完全不同——前者靠更好的代码，后者靠重新定义产品边界。

### 5.2 系统谦逊是可行的第三条路

文生图的"AI 替你画好"和传统工具的"你全手动操作"之间，存在第三条路：**AI 不假装有把握，用户不期待一次成型**。

Say2Draw 的"草图起步，一起打磨"模式在 PR17-PR25 期间被验证有效：用户看到画布上有东西（哪怕粗糙）后，就有了具体的锚点可以描述怎么改。从模糊到具体的过程不需要 AI 在第一步就猜对，只需要 AI 每次只做一个小改动，逐步逼近。

### 5.3 PR 拆分粒度是架构意识的训练场

从 PR0-PR16 的"按功能切"（语音/解析/绘图各自独立 PR），到 PR24/PR25 的"按架构层切"（模型层 vs 视图层），这个演化不是刻意追求，而是在反复调试中自然涌现的。

按功能切的问题：一个功能改动常常同时触及算法逻辑（"怎么画"）和 UI 展示（"让用户看到什么"），混在一起难以独立测试和回退。按架构层切解决这个问题：PR24 只动模型，PR25 只动视图，各自独立验证。

### 5.4 供应商去风险不是过度工程

DeepSeek 的 structured output 兼容性问题在 PR9-PR16 期间反复出现，每次修复都是针对特定 provider 的 workaround。PR17 果断切换到 Mimo，同时保持 OpenAI-compatible 接口抽象层，让未来的 provider 切换成本降到最低。

**教训**：AI 应用的 provider 绑定风险不是理论上的，而是会在项目进行中实际发生。越早建立 provider-agnostic 接口层，后期迁移成本越低。

## 6. Future Directions

如果继续投入 Say2Draw，最值得做的三个方向：

### 6.1 画布状态持久化

当前画布状态仅存于 React state，刷新即丢失。持久化最简单的路径是 LocalStorage + JSON 序列化（储存 shapes + sketchStrokes），不需要引入数据库。这会让用户能保存和回溯自己的创作过程。

### 6.2 多轮对话上下文记忆

当前每次语音指令独立处理，不保留上下文。如果用户说"把它变成蓝色"，系统需要记得"它"指代的是上一个生成的图形。短程上下文（最近 2-3 轮操作）可以用前端 state 追踪，不需要额外的 LLM 调用。

### 6.3 更丰富的本地即时调整

PR21 已支持基础的移动/缩放/颜色调整（正则匹配，零 API 成本）。可扩展方向：旋转、翻转、复制、图层排序。这些操作的共同特点是纯几何变换，不需要语义理解，可以实现为本地即时响应。
